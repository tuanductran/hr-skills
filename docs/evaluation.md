# Evaluation framework

> Phase 4.4 of the [roadmap](ROADMAP.md) — deterministic quality evaluation for the Skill Planner and Workflow Runtime.

## What it is

The Evaluation framework runs a set of representative planning scenarios through the real [Skill Planner](planner.md) and [Workflow Runtime](runtime.md), compares the outcome against a committed golden fixture, and reports deterministic quality metrics.

It is responsible for:

- Running evaluation datasets against the real Skill Registry
- Detecting regressions by diffing each run against a committed golden fixture
- Measuring planner quality (capability matching, skill selection, execution ordering)
- Measuring workflow quality (dependency correctness, execution success)
- Producing a deterministic benchmark report

The Evaluation framework is **NOT** responsible for:

- Intent analysis, capability matching, or skill selection — that's the Planner
- Executing steps or propagating context — that's the Runtime
- Authoring or curating skill content — that's the skills themselves

It only observes and scores the existing pipeline. It calls the exact same functions the CLIs use (`generateExecutionPlan`, `validateExecutionPlan`, `executeWorkflow`) rather than reimplementing planning or execution logic, so it evaluates real behavior, not an approximation of it.

## Architecture

```text
eval/datasets/*.json (intents)
        |
        v
  runCase(case, registry)
        |
        |-- generateExecutionPlan   (Planner)
        |-- validateExecutionPlan   (Planner validation)
        |-- executeWorkflow         (Runtime, stub step executor)
        |
        v
  GoldenCaseResult (skillIds, capability coverage, validity, workflow status)
        |
        v
  diffAgainstGolden(actual, eval/golden/*.golden.json)
        |
        v
  EvaluationReport (per-case results + aggregated QualityMetrics)
```

| Component | File | Responsibility |
|---|---|---|
| `EvaluationDataset` loader | `src/evaluation-datasets.ts` | Discovers and loads dataset JSON files, and loads/saves golden fixtures. |
| `runCase` / `runDataset` | `src/evaluate.ts` | Runs one case (or a whole dataset) through the Planner and Runtime and captures a `GoldenCaseResult`. |
| `diffAgainstGolden` | `src/evaluate.ts` | Compares an actual result against its golden fixture entry and returns the list of differing fields. |
| `computeQualityMetrics` | `src/evaluate.ts` | Aggregates per-case results into the five `QualityMetrics` ratios. |
| `runEvaluation` / `toGoldenFixture` | `src/evaluate.ts` | Orchestrates a full dataset run into an `EvaluationReport`, and converts a report into a fixture for `--update-golden`. |
| CLI (`bun run evaluate`) | `src/run-evaluation.ts` | Runs every dataset, prints a report, writes `eval-report.json`, and exits non-zero on regressions or invalid plans. |

All shared type definitions (`EvaluationCase`, `EvaluationDataset`, `GoldenCaseResult`, `GoldenFixture`, `EvaluationCaseResult`, `QualityMetrics`, `EvaluationReport`) live in `src/types.ts`, alongside the Planner's and Runtime's types.

## Dataset format

Datasets live in `eval/datasets/*.json`. Each file is an `EvaluationDataset`:

```json
{
  "name": "planning-scenarios",
  "description": "...",
  "cases": [
    {
      "id": "recruiting-interview-questions",
      "description": "A single, narrow recruiting request.",
      "intent": "create interview questions for a senior manager",
      "category": "recruiting"
    }
  ]
}
```

A dataset stores only the **input** — the case's `intent` (plus a stable `id`, a human `description`, and a `category` used to group results in reports). It deliberately does not hand-author the *expected* output: guessed expectations drift from what the Planner and Runtime actually produce as the registry and matching logic evolve. Expected output instead comes from a golden fixture.

## Golden fixtures

Golden fixtures live in `eval/golden/<dataset-name>.golden.json` and are generated by actually running the pipeline once against the real Skill Registry, then committed to the repository. Each fixture is a `GoldenFixture`:

```json
{
  "dataset": "planning-scenarios",
  "generatedAt": "2026-07-24",
  "results": [
    {
      "caseId": "recruiting-interview-questions",
      "skillIds": ["hr-interviewing", "hr-candidate-assessment"],
      "matchedCapabilities": 1,
      "totalCapabilities": 1,
      "planIsValid": true,
      "workflowStatus": "completed"
    }
  ]
}
```

A `GoldenCaseResult` captures only the fields that are meaningful to compare across runs:

- `skillIds` — the Planner's selected skills, **in execution order** (so a re-ordering counts as a change, not just a different selection)
- `matchedCapabilities` / `totalCapabilities` — capability coverage
- `planIsValid` — whether `validateExecutionPlan` reported zero errors
- `workflowStatus` — the Runtime's final status (`completed` or `failed`) when the plan is executed with the same stub step executor `execute-plan.ts` uses

It intentionally excludes incidental detail (rationale strings, timestamps, full event/trace logs) that would make the fixture noisy and produce false-positive regressions unrelated to planning or execution quality.

## Quality metrics

`computeQualityMetrics` aggregates every case's result into five 0.0-1.0 ratios:

| Metric | Definition |
|---|---|
| `capabilityMatchingAccuracy` | Total matched capabilities across all cases ÷ total requested capabilities. |
| `skillSelectionAccuracy` | Fraction of cases whose selected skill **set** exactly matches the golden fixture. |
| `executionOrderingAccuracy` | Fraction of cases whose selected skill **sequence** exactly matches the golden fixture (a set match with a different order still counts against this metric). |
| `dependencyCorrectness` | Fraction of cases with zero `validateExecutionPlan` errors (no circular dependencies, dangling references, or order violations). |
| `workflowSuccessRate` | Fraction of cases whose Runtime execution finished with status `completed`. |

Every ratio is defined for an empty case set (denominator 0 maps to a score of 1 — vacuously satisfied), so metrics never produce `NaN`.

## Regression detection

For each case, `diffAgainstGolden` compares the current run's `GoldenCaseResult` field-by-field against the committed golden entry and returns the names of any fields that differ (`skillIds`, `matchedCapabilities`, `totalCapabilities`, `planIsValid`, `workflowStatus`). A case with no committed golden entry yet (a brand-new case) reports no regressions — there is nothing to regress against.

`EvaluationReport.regressedCaseIds` lists every case with at least one regressed field. The CLI exits non-zero when any case regresses or fails validation, which makes `bun run evaluate` suitable as a CI gate.

Because the Planner and Runtime are both pure/deterministic (no wall-clock timestamps, no randomness — see [`docs/planner.md`](planner.md) and [`docs/runtime.md`](runtime.md)), two runs against an unchanged registry always produce byte-identical `GoldenCaseResult`s. This is what makes golden-fixture comparison a reliable regression signal rather than a source of flaky failures.

## Benchmark execution

```bash
bun run evaluate
```

This builds the Skill Registry, loads every dataset in `eval/datasets/`, runs each case, diffs it against the matching `eval/golden/*.golden.json` fixture, prints per-case results and the aggregated quality metrics, and writes the full report to `eval-report.json`. Exit code is `0` when every case is regression-free and every plan is valid, `1` otherwise.

To regenerate golden fixtures after an intentional change to the Planner, Runtime, or skill content (a new skill, a renamed capability, and so on), run the script directly from `packages/hr-skills-build` (the `bun run evaluate` root alias does not forward flags through Turborepo):

```bash
cd packages/hr-skills-build
bun src/run-evaluation.ts --update-golden
```

Review the diff on `eval/golden/*.golden.json` in the resulting commit like any other code change — a `skillIds` change there is exactly the signal that a golden fixture would otherwise catch as a regression, so it deserves the same scrutiny.

## Adding new evaluation cases

1. Add a case to an existing dataset (`eval/datasets/<name>.json`), or create a new dataset file — any `*.json` file in `eval/datasets/` is auto-discovered by `loadAllDatasets`.
2. Give the case a stable, descriptive `id` (unique within its dataset), the `intent` to evaluate, a `category` for grouping, and a short `description`.
3. Run `bun run evaluate --update-golden` to generate the golden entry for the new case from the real pipeline.
4. Inspect the resulting `eval/golden/*.golden.json` diff — confirm the selected skills, capability coverage, and validity are what you expect for the scenario, since this is what future runs will be held to.
5. Commit the dataset and golden fixture changes together.

## Testing

- `test/evaluate.test.ts` — unit tests for `runCase` (determinism, unmatched intents), `diffAgainstGolden` (field-level regression detection), and `computeQualityMetrics` (each metric in isolation, including the empty-set and all-pass cases).
- `test/evaluation-datasets.test.ts` — dataset/golden fixture loading, discovery, and the "no fixture yet" case.
- `test/evaluation-integration.test.ts` — runs `planning-scenarios` against the real Skill Registry and the committed golden fixture; fails with a per-case, per-field diff if anything regresses. Skippable with `SKIP_INTEGRATION=true`, matching `planner-integration.test.ts`.

Run with:

```bash
bun run test
```
